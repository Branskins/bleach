# Solving HackerRank challenges

## Workflow (per problem)

1. Navigate to the challenge (`https://www.hackerrank.com/challenges/<slug>/problem`) and read the problem with `get_page_text`, not a screenshot — text extraction is far cheaper than repeated screenshots and captures the input/output spec verbatim. If numbers/ranges look stripped from the extracted text (rendered as math spans, common on constraint lines), take one screenshot to confirm the exact figures before coding — don't guess at bounds.
   - **Don't guess the slug from the title** (e.g. `python-triangle-quest` / `triangle-quest` for "Triangle Quest") — HackerRank's actual slugs are often unrelated to the title (the real one was `python-quest-1`), and a wrong guess 404s. If a guessed slug 404s, navigate to `https://www.hackerrank.com/domains/python?filters%5Bsubdomains%5D%5B%5D=<subdomain-key>` (subdomain keys are visible as query params when clicking a subdomain filter on the domain page, e.g. `py-math`) and use `find` on the results list to get the real `href`, rather than guessing more URL variants.
2. Write the solution locally first: `problems/hackerrank/python/<slug>.py` (a small pure function plus an `if __name__ == "__main__":` block doing the actual stdin/stdout wiring) and `problems/hackerrank/python/test_<slug>.py`. Run `pytest problems/hackerrank/python/test_<slug>.py -q` before touching the browser — this catches logic bugs for free, without spending a HackerRank submission attempt on a bug that pytest would have caught instantly.
3. Reproduce the verified logic in HackerRank's browser editor (see editor-interaction rules below).
4. Click **Run Code** first and confirm the sample test cases pass via `get_page_text` (look for "Congratulations! You have passed the sample test cases"). Only click **Submit Code** once samples pass — submitting broken code wastes the interaction and muddies the point tally.
5. After Submit Code, always re-read the page with `get_page_text` before believing it worked — see "Verifying success" below.
6. Update `problems/hackerrank/python/unsolved.md`: flip `[ ]` to `[x]` for that row and turn the title into a markdown link to the local solution file, e.g. `[x] [Athlete Sort](athlete_sort.py)` — only after step 5 confirms the points actually landed. The link target is just the filename (unsolved.md lives in the same directory as the solutions, so no path prefix is needed).

## Editor interaction rules (HackerRank's CodeMirror instance)

These were learned the hard way across a 12-problem session — skipping any of them silently produces broken or unsubmitted code.

- **The editor needs a double-click to focus.** The first `left_click` on a code line often lands visually but doesn't give the editor real keyboard focus — the next `type`/`key` action is silently swallowed and the code stays unchanged. Click the target line twice before sending `End`/`Return`/`type`. Verify with a screenshot after typing rather than trusting the action reported success — a click landing outside the true text-input hitbox produces no error, just no effect.
- **`\n` inside a `type` string does not reliably produce a newline in this editor.** Typing `"line one\nline two"` in one call can leave the editor showing only the first line, unchanged. Use explicit `key: "Return"` presses between lines instead of embedding `\n` in the typed string.
- **Auto-indent is inconsistent and cannot be trusted blindly.** Sometimes pressing Return after a line ending in `:` bumps the next line's indent by one level as expected; other times a Return after a *non-colon* line still fails to preserve the previous line's indent. Relying on it produced a real `IndentationError: expected an indented block after 'for' statement` mid-session. Two ways to stay safe:
  - **Prefer single-line compound statements** (`if op == "insert": lst.insert(...)`, `elif op == "print": print(lst)`, `for i in range(n): print(i, end="")`) over multi-line blocks whenever the body is one statement — this sidesteps the indent question entirely since there's no nested line to get wrong.
  - When a real nested block is unavoidable, **don't trust the auto-indent** — after typing, screenshot and check every line's leading whitespace matches its intended nesting level. Fix a misindented line by clicking it, pressing `Home` (goes to the first non-whitespace character), and typing/deleting spaces directly. Avoid Backspace-based dedent on an empty auto-indented line — it has been observed to delete into the *previous* line's actual content instead of just the leading whitespace, corrupting a working line. Always screenshot-check after any indent fix.
- **Long logical lines soft-wrap across 2-3 visual rows** — this is cosmetic only; don't mistake a wrapped single statement (e.g. a long list comprehension) for multiple lines when reading a screenshot.
- **The language dropdown sometimes defaults to "Pypy 3"** on a fresh problem load — switch it to "Python 3" before typing, since the starter stub and this project's conventions assume CPython/Python 3 semantics. Some older Python-domain challenges (e.g. Incorrect Regex) don't offer a "Python 3" option at all — only Pypy 2/Pypy 3/Python 2. Pick Pypy 3 for standard Python 3 syntax; only drop to Python 2 if the judge's older stdlib behavior is actually required (see interpreter-version mismatches below), and remember Python 2 needs `raw_input()` instead of `input()` for reading strings (`input()` in Python 2 evaluates the line as a Python expression, which is wrong for arbitrary string input).
- **Triple-click on the target line to select-and-replace it — this is now the default, reliable method** (see below); a plain single `left_click` positioned at the end of a short stub line (e.g. a bare `print` placeholder awaiting the rest of the call) can silently fail to place the cursor where expected, leaving a subsequent `type` a no-op. When single-click-then-End doesn't visibly change anything, retry with `double_click` on the word itself (e.g. on `print`) before pressing `End` — this reliably re-establishes real focus and cursor position even on very short lines where triple-click would select the whole line instead of letting you append.
- **Typing directly into a genuinely blank stub line (no word to `double_click` on) can drop the line's indentation** — confirmed on Athlete Sort: a blank line inside an `if __name__:` block, at the same 4-space indent as its siblings, came out at column 0 after `double_click`+`type` on that empty line. Always screenshot immediately after typing into a blank line and compare its indentation against the sibling statement directly above it; fix with `Home` + typing the missing spaces (blank lines have no real content for `Home` to treat as "smart", so a plain `Home` here does go to column 1).
- **`Home` is "smart home", not column-1 home.** The first `Home` press moves to the first non-whitespace character on the line, not column 1 — so `Home` then `shift+End` selects everything *except* the leading indentation, and a follow-up `type` only replaces the line's content while leaving stray original whitespace behind (or, worse, leaves the old comment/text partially in place when the click landed mid-line and `shift+End`/`shift+Home` only captured part of it). This produced repeated garbled-line bugs (duplicated fragments, missing characters, edits landing on the wrong line) across a session. Prefer **triple-click on the line** to select it fully (works reliably even with leading indentation) before typing a replacement — this was the one selection method that consistently worked. If a line still ends up mis-indented after triple-click-replace, fix it by clicking at the true column-0 x-coordinate (same x as an unindented line like `def ...` or `if __name__`) rather than pressing `Home`.
- **Either the `if __name__ == '__main__':` block OR the top import/shebang block can be read-only — it varies per problem, and it's not always the same half.** Most function-based stubs lock the main block (edits inside it, e.g. changing `func(n)` to `print(func(n))`, appear to succeed in a screenshot immediately after typing but silently fail to persist — Run Code shows the original unmodified boilerplate). But on at least one problem (Company Logo) it was the reverse: the top `import math / import os / ...` block was locked (typing `from collections import Counter` there produced zero change, confirmed via the `Line: X Col: Y` status readout not advancing), while the `if __name__` main block was fully editable, including adding new imports inside it. Don't assume which half is locked — after your first edit in a stub with two sections, re-read the page text (or check the status-bar column count moved) before writing more code on the assumption it "took". If the main block won't accept a `print(...)` wrapper, move the print inside the function instead (per the rule above); if the top block won't accept an import, just add the import statement directly inside the main block/wherever the editable region is — Python allows imports anywhere a statement is valid.
- **Switching the Language dropdown regenerates the editor's starter stub and discards whatever you've typed.** This is sometimes useful on purpose (e.g. switching from Pypy 3 to Python 3 handed back the exact two "first two lines" stub the problem's Note referred to, which had gone missing under Pypy 3) but it means: switch language *before* writing any code, not after — a language switch mid-edit throws away the current buffer.
- **When repeated edits to one line produce corrupted/duplicated text**, stop patching incrementally — click into the editor, `ctrl+a` then `Delete` to clear it completely (verify the editor is actually empty via screenshot; `ctrl+a` occasionally only selects a sub-block, not the whole document, on this CodeMirror instance), then retype the entire solution fresh line-by-line. This is more reliable than chasing exactly where a bad selection landed.
- **When appending text via `End` to a stub line right above a blank stub line, double-check which line the click actually landed on before typing — long appended text causes that line to wrap into extra visual rows, and a *second* `double_click`+`End` aimed at "the next stub line" by y-coordinate can land on one of those wrapped visual rows instead of the real next logical line, silently merging your new text onto the end of the previous statement.** This happened on Classes: Dealing with Complex Numbers (a multi-method class stub): appending a return statement to `__init__` made it wrap to 3 rows, and the next `double_click` (aimed at the y-coordinate of the blank line below `__init__`, which was actually where `__add__`'s def line now rendered) landed mid-way through the wrapped `__init__` line instead, merging `def __add__(self, no):` onto it. Screenshot after *every* single-line append (not just at the end of a batch) when stub lines are short and closely spaced — catching a wrap-merge one edit later is far cheaper than untangling it after 4 more edits landed on now-shifted coordinates. To recover from a confirmed merge: click precisely before the merged-in text (zoom in on the region first to find the exact pixel boundary), `shift+Down shift+End` to select through to the true end of the accidentally-merged line, and `Delete` — then re-add the removed statement as its own new line using `double_click`+`End` on its real, now-correct line. Also: text that appears to go missing from the *start* of every visible line after a long line was typed (e.g. `import math` rendering as `mport math`) is very likely just horizontal auto-scroll from that long line, not real deletion — confirm with `ctrl+Home` before assuming corruption.
- **Interpreter-version *and* interpreter-implementation mismatches between local dev Python and HackerRank's judge are real and will fail a submission that passed local pytest.** Confirmed cases: (1) `ZeroDivisionError` for `int // int` by zero says "integer division or modulo by zero" on older interpreters but just "division by zero" on newer ones — if a problem's expected output hardcodes an exception message, catch that specific exception and hardcode the expected string rather than relying on `str(e)`. (2) Python's `re` module used to reject "multiple repeat" patterns like `.*+` as invalid but modern CPython/PyPy accept them — if `Run Code` shows your regex-validity check disagreeing with the expected sample output, the judge is likely running an older engine; switching the language dropdown to the oldest available variant (e.g. Python 2) can restore the expected stricter behavior. (3) **PyPy's `set.pop()` does not remove elements in the same order CPython's does**, even for identical small-int sets — a problem whose expected trace assumes CPython's pop order (e.g. `set(range(1,10)).pop()` removing `1` first) threw a `KeyError` under "Pypy 3" but passed immediately after switching to plain "Python 3". If a set/dict ordering-sensitive problem fails under Pypy with a KeyError or wrong-order output but your logic looks right, switch to CPython (`Python 3`/`Python 2`) before debugging further — it's very likely the runtime, not the code. Don't assume local pytest passing means the judge will agree on error-message, regex, or set-iteration-order edge cases — treat the sample test case's actual expected output as the source of truth over local intuition when they conflict, and treat "switch the language variant" as a legitimate fix, not just a fallback. (4) **Not every interpreter mismatch is fixable by switching languages** — see the Tuples hash quirk below, where both "Python 3" and "Pypy 3" gave the identical (wrong) answer, because HackerRank's judge itself, not just the local dev machine, has moved to a newer hash algorithm than the one used to generate the stored expected output. Try the language-switch fix first (it's cheap), but if the *same* wrong output reappears under a different language, stop switching languages and go looking for an algorithm/version mismatch instead.
- **Full-buffer clear-and-retype (`ctrl+a` → `Delete` → retype everything) is the least reliable editing method on this CodeMirror instance and should be a last resort, not the default.** Across a 10-problem batch it repeatedly no-op'd (screenshot showed the exact same stub after the key sequence, as if never pressed), or silently partial-selected so the retype landed concatenated onto old code (e.g. `print(s.swapcase())def swap_case(s):` — the new text glued onto leftover old text with no newline between). **Always screenshot immediately after `ctrl+a` and confirm the whole buffer is highlighted blue before pressing Delete** — if nothing is highlighted, click squarely inside the visible text (not in blank space below it) and retry `ctrl+a` before proceeding; don't just chain `ctrl+a`→`Delete`→`type` blind. **Prefer a targeted edit instead whenever the given stub already contains most of the needed scaffolding** (a bare `def foo(...): return` / `# write your code here` placeholder, or a fill-in-the-blank stub like Text Alignment's `______`): `double_click` directly on the placeholder word/line, confirm via screenshot that only the intended token is highlighted (not a same-named neighbor — see below), then `type` the replacement. This is strictly safer because the surrounding stub's indentation and structure are never touched.
- **If a full clear-and-retype does go wrong (corrupted/duplicated/concatenated text), don't hand-patch it — click the "Reset Code" button** (circular-arrow icon in the editor toolbar, left of the fullscreen icon) and confirm the "Attention: Reset Code" dialog with **Yes**. This restores the exact pristine starter stub in one click and is far faster than tracing where a bad selection or dropped keystroke landed. Re-verify with a screenshot that the stub is back to its original state before re-editing.
- **`double_click` on a fill-in-the-blank token (e.g. `______`) reliably selects the whole run of underscores as one unit** (confirmed on Text Alignment, which has 8 separate blanks across 5 lines) — this is a fast, low-risk way to fill in `rjust`/`ljust`/`center`-style blanks without retyping the surrounding scaffolding. But **the click coordinate must land on the blank itself, not on a same-looking or adjacent identifier** — on a line like `print((c*thickness).______(thickness*2))`, a double-click aimed slightly wrong can land on the word `thickness` instead of the blank next to it, and the typed replacement then silently overwrites a real identifier instead of the placeholder (produces a subtle bug: the code still runs, but with wrong values, not a syntax error). Always **zoom into the target line first** (`computer` action `zoom` with a tight region around just that line) to read the exact pixel span of the blank before double-clicking, and **screenshot immediately after the double-click** to confirm the highlighted selection is the blank (a run of `_` characters) and not a word — if it's wrong, `ctrl+z` immediately before typing anything.
- **Rapid-fire batches of many `Return`/`Backspace` key presses can silently drop individual keystrokes**, producing two lines concatenated with no newline between them (`x -= 1 << 64return -2 if x == -1 else x`) even though the tool result reports every key press as "succeeded." This happened consistently enough across one multi-line indented function that the safest fix was to **stop fighting the indentation entirely: rewrite the logic as a single flat, zero-indentation script** using `functools.reduce` with a lambda, generator expressions (`sum(1 for i in ... if ...)`), or list comprehensions in place of `for`/`if` blocks with bodies. A flat script has no `Return`-after-colon or dedent-via-Backspace steps to drop, so this class of bug cannot occur. Prefer this over multi-line indented blocks by default when the stub is fully open (not a fill-in-the-blank), even beyond the "single-line compound statement" case already covered above — e.g. turn a whole helper function into one expression assigned to a variable, then `print()` it.
- **When a function's stub docstring comment says "expected to return a STRING" but the actual `if __name__ == '__main__':` block calls it as a bare statement (`print_full_name(first_name, last_name)`) without wrapping it in `print(...)`, the function must `print(...)` internally, not `return`** — the comment can be generic boilerplate left over from a different template and does not override what the actual driver code does with the return value. Always expand/read the real `if __name__` block (not just the doc comment above the function) before deciding whether to `return` or `print` from inside a stub function.
- **Mouse-drawn selections (`triple_click`, `shift+click`, `ctrl+a`, `ctrl+shift+End`) reliably highlight blue in a screenshot but `Backspace`/`Delete`/`type` immediately afterward can be a complete no-op against that selection** — confirmed repeatedly on Map and Lambda Function: a `shift+click` spanning all 12 lines showed full blue highlight and a correct `Line: 12 Col: 41` status readout, then `Backspace` left the buffer byte-for-byte unchanged with the status readout reset to `Line: 1 Col: 1`. This is a deeper version of the "full-buffer clear-and-retype is unreliable" rule above: it's not just `ctrl+a` that's flaky, *any* selection-then-delete is unreliable, regardless of how it was drawn or how small it is (even a single-line `triple_click`+`Backspace` failed the same way). **The one primitive that reliably works is deleting from a bare caret with no selection**: `click`/`Home`/`End` to position the caret (verify via the `Line: X Col: Y` status text matching your expectation), then `Backspace` or `Delete` — a single press, or the `computer` tool's `repeat` parameter for many presses in one call, both work correctly and predictably consume characters from that exact position. Rewrite any "select this range and replace it" instinct as "put the caret at one end and press Backspace/Delete `repeat` times" instead. Also: `repeat`-based deletion safely stops the instant it reaches a locked/read-only boilerplate boundary (see the `if __name__` lock rule below) rather than skipping past it — extra repeats beyond what's needed are harmless.
- **Flattening a method body to one line (`def foo(self, x): print(x); [do(y) for y in x]`) does NOT eliminate the need to indent the `def` line itself under its enclosing `class`.** Typing several `def ...: <one-liner-body>` lines in a row at column 0 right after `class Foo(Base):` + `Return` produces `IndentationError: expected an indented block after class definition` — the flattening trick (used elsewhere in this skill to avoid *nested* for/if indentation inside a function body) only removes the *inner* block's indentation risk; the outer `class`/`def` header still requires its immediate body to be indented one level, even when that immediate body is itself a single flat line. Confirmed on HTML Parser - Part 1: three `def handle_*` one-liners typed at column 0 directly under `class P(HTMLParser):` compiled with an IndentationError; fix was to click `Home` on each `def` line and prepend exactly 4 spaces, then re-run. When writing a class with several one-line methods, type the class header, then prefix every method-def line with the class's indent level (4 spaces) before typing the rest of that line — don't rely on auto-indent carrying over correctly across a `Return` between two sibling one-liner methods (it did not in this case).
- **A `Backspace`/`Delete` `repeat` count that overshoots the leading whitespace on a line will keep eating backward into the *previous* line's trailing characters, silently merging two lines and corrupting both** — confirmed correcting a wrongly-indented line 9 (`P().feed(html)`): after typing 4 extra spaces then sending `Backspace` with `repeat: 4` to undo them, the actual indent turned out to be 3 not 4, so the 4th backspace deleted the newline and continued deleting `'`, `n` from the end of line 8's `'\n'` string literal, merging the two lines into `...+ '\P().feed(html)` with no error or warning. Always screenshot (or re-check the `Line: X Col: Y` readout) immediately after any `repeat`-based Backspace/Delete used to undo an indent change, rather than assuming the repeat count you added is the exact count needed to remove — asymmetric add/remove counts are an easy off-by-one to miss silently.
- **Python's indentation rule is exact-match, not "more than parent"** — sibling statements in the same block (e.g. two `def` lines directly under one `class`) must have *identical* leading whitespace, not just "more indented than the class line." Typing a fresh line into a stub's existing blank line inherited whatever auto-indent was already on that blank line, then a manually-typed leading `"  "` on top of it produced 3-4 spaces where a sibling `def` above it had exactly 2, causing `IndentationError: unexpected indent` (HTML Parser - Part 2, `handle_data` vs `handle_comment`). Don't assume a leading-space count you type is the *total* indent — the target line may already carry auto-indent before your cursor. After typing/fixing any indent, click `Home` (smart-home lands on the first non-whitespace char) and check the `Col` readout against the sibling line's `Col` at the same position — they must match exactly, not just both be "indented". Fix a mismatch with `Backspace` (deletes the whitespace *before* the cursor) — using `Delete` at the smart-home position instead deletes the first real character of the line, not whitespace, since there's no whitespace after that cursor position to remove.
- **When a multi-line class-with-methods stub keeps producing indent mismatches, the fastest fix is `ctrl+a` → `Delete` → retype from scratch checking `Home`+`Col` after every single `Return`, not patching the tangled result in place.** On Detect HTML Tags, chasing an indent bug through several rounds of `Backspace`-with-`repeat` produced cascading line-merge corruption (see the rule above) that took multiple recovery steps to unwind. Clearing and retyping took one pass: type a line, `Return`, `Home`, screenshot/check `Col` — if it's `Col: 1` this editor is *not* auto-indenting after that particular `:`-ended line (confirmed on this problem — unlike others in this same session where auto-indent did fire), so type the exact leading spaces yourself for every subsequent sibling line at that level, and expect `Col: 1` again (no compounding) after each of those single-line-body `def` statements. Don't assume auto-indent behavior carries between problems — verify it fresh each time with one check right after the first `:`+`Return`.
- **`get_page_text` can render a long wrapped code line's visual rows out of their true logical order** (e.g. a line that soft-wraps 2 rows showing the second row's text *before* a shorter line that visually appears between them on screen) — this is a text-extraction artifact of reading wrapped `<div>` rows, not a real bug in the code. Don't "fix" code because `get_page_text`'s line order looks scrambled; trust the `Congratulations!`/`Wrong Answer` verdict and the `Your Output` vs `Expected Output` blocks over the raw extracted line ordering of the code panel itself.

## Known problem-specific quirks

- **Tuples (`python-tuples`): `hash(tuple(...))` will not match HackerRank's stored expected output on *either* "Python 3" or "Pypy 3"**, because CPython's tuple-hash algorithm changed in 3.8 (old FNV-style combine → new xxprime-based combine) and HackerRank's expected output was generated with the old algorithm, while both language options on the judge now run the new one. Switching languages (the usual fix for interpreter mismatches, see above) does *not* help here — both give the identical wrong value. The fix is to reimplement the old algorithm directly, as a flat one-liner to sidestep indentation risk:
  ```python
  import functools
  MASK = (1 << 64) - 1
  n = int(input())
  values = list(map(int, input().split()))
  init = (0x345678, 1000003, len(values))
  state = functools.reduce(lambda s, item: (((s[0] ^ hash(item)) * s[1]) & MASK, (s[1] + 82520 + 2*(s[2]-1)) & MASK, s[2]-1), values, init)
  x = (state[0] + 97531) & MASK
  x = x - (1 << 64) if x >= (1 << 63) else x
  print(-2 if x == -1 else x)
  ```
  Verify locally against the known sample expected value (`3713081631934410656` for input `2\n1 2`) with a plain `assert`, not a tautological test that just recomputes `hash()` and compares it to itself (that passes trivially on any interpreter and proves nothing about matching HackerRank's judge).

## Verifying success

Never assume a click on Submit Code worked — confirm every time:
- Re-read the page with `get_page_text` after a `wait` of 2-3 seconds.
- Look for `"Your <Challenge Name> submission got N.NN points!"` and a higher `Points: X/Y` tally than before submitting.
- If the points total is unchanged and the editor still shows an editable (not "Congratulations") state, the click didn't register. Common causes: an interrupted/rejected prior tool call left the actual click un-executed, or a `navigate` call reset scroll position so a coordinate-based click landed on the wrong element after the page re-flowed. Re-locate the button with `find` (query: "Submit Code button") and click by `ref` rather than raw coordinates — refs survive layout shifts that coordinates don't.

## Cost/context management

- Prefer `get_page_text` over `screenshot` for reading problem statements and submission results — it costs a fraction of the tokens.
- Reserve screenshots for moments that need visual confirmation: checking the editor's actual text content after typing, and locating click targets right after a fresh page load or scroll.
- Batch sequential browser actions (click, key, type, screenshot) into one `browser_batch` call rather than one tool call each — this cuts round-trips substantially over a multi-problem session.
- This work is genuinely token-heavy. Pace it in batches of roughly 5-10 problems per session rather than trying to clear the whole `unsolved.md` list at once — context usage from browser-automation tool output climbs fast, well before real problem-solving difficulty becomes the limiting factor.
